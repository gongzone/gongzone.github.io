## 1. 개요

**CSR(Client Side Rendering)**, **SSR(Server Side Rendering)**, **SSG(Static Site Generation)** 은 웹 개발을 하면서
쉽게 접할 수 있는 용어들이다. 이 세가지 방식은 렌더링 시 필요한 데이터를 어느 타이밍에 주입할 것이냐에 따라 구분되어 진다.
즉, 클라이언트(브라우저)에게 그 책임이 있다면 CSR, 서버에게 그 책임이 있다면 SSR, 애플리케이션을 빌드하는 순간에 그 책임이 있다면
SSG로 나눌 수 있다. 현대 리액트 프레임워크들(Next.js, Gatsby, Remix)은 하나의 페이지 렌더링 방식을 고집하지 않는다.
세 방식 모두 각자 장단점이 있기에 장점들을 엮어 솔루션으로서 제공하려는 움직임이 있어왔다.

## 2. CSR (Client Side Rendering)

**URL(Uniform Resource Locator)** 은 특정 서버의 위치를 표준화한 이름이다. 우리가 URL에 접속한다는 것은 그 특정 서버로부터 필요한 자원을
받아오겠다는 것을 의미한다. 일반적인 웹 사이트를 접속했을 때 여러 자원들(html, css, js, font, image ...)이 우리 브라우저를 통해 들어오겠지만, 그 중 가장 중요한 것은 웹사이트의
마크업 구조인 HTML 파일이다.

**CSR**의 경우 HTML 파일이 거의 텅빈 상태로 들어오게된다. 따라서 우리가 처음 맞닿뜨리는 화면은 빈 화면일 뿐일 것이다.
CSR 방식을 사용하면 이 빈 화면을 채우는 것이 클라이언트의 몫이 된다. 어떻게? **자바스크립트를 통하여 동적으로 DOM 구조를 그린다.**

즉, 하나의 빈 HTML 파일을 자바스크립트를 이용하여 컨트롤 하는 것이다. '/'로 접속했을 때는 루트 페이지에 관한 DOM 구조와 데이터를 넣고,
'/about' 페이지로 접속했을 때는 about 페이지에 관한 정보를 렌더링하는 방식이다. 초기에 하나의 HTML 파일을 받아오기만 하면 되기 때문에,
이러한 방식으로 구현된 애플리케이션을 **SPA(Single Page Application)** 이라고 부른다.

<Callout>
  **장점**  
  1. 페이지 이동 간, 깜박임 현상이 발생하지 않기 때문에 더 나은 UX를 제공한다.
  2. 서버가 부담해야할 요청이 적다.

  **단점**
  1. 초기 로딩 속도가 느리다. (자바스크립트 파일 용량 큼)
  2. SEO에 취약하다. 
</Callout>

<Callout kind="info">
  **추가 정보**
  - 리액트 프레임워크를 사용하지 않더라도 순수 리액트 만으로도 SSR 환경을 구성할 수 있다.
  - 리액트가 CSR 형식으로 구현되었을 때, 초기 로딩 속도를 높이기 위해 코드 분할(Code Splitting)을 사용할 수 있다.
  - 구글의 경우 크롤링에 있어 CSR인 경우에도 SEO 취약 문제를 어느정도 해결했다고는 하나 완벽하진 않다는 것이 대부분의 의견이다. 
</Callout>

## 3. SSR (Server Side Rendering)

**URL**을 통해 페이지 요청이 들어오면, **SSR**의 경우 서버에서 필요한 데이터를 가져와(fetching) HTML에 주입하여 완전한 구조의 파일을 반환한다.
CSR의 경우와 다르게 각각의 페이지가 독립적으로 존재하게 되어, 페이지 이동시마다 서버에게 요청을 보내야 한다.

페이지별로 자원을 불러오기에 초기 로딩속도에 있어 이점이 존재하며, 완전한 마크업 구조를 가졌기에 SEO 측면에서 더 나은 성능을 보인다.

<Callout>
  **장점**  
  1. 초기 로딩 속도가 빠르다. (자바스크립트 파일 용량 적음)
  2. SEO 성능이 좋다.

  **단점**
  1. 페이지 이동간 새로운 자원을 받아오기에, 깜박임 현상이 발생한다.
  2. 서버가 부담해야할 요청이 많다.
</Callout>

## 4. SSG (Static Site Generation)

**SSG**는 앞선 두 방식과는 전혀 다른 전략을 취한다. 런타임에 복잡한 렌더링 로직을 수행하는 것이 아니라 말그대로 **그저 보여주는 것**이다.
즉, 빌드 타임에 완전한 파일을 만들어두고 요청이 들어오면 그 파일을 그 자체로 클라이언트 쪽으로 보내주는 것이다.

따라서 페이지 이동시, 브라우저는 파일을 받아오기만 하면 되기 때문에 웹사이트가 더 빠르게 동작할 수 있다.
또한 SEO 측면에서도 문제가 없다. 

그러나 단점으로는 컨텐츠가 동적으로 많이 변화하는 경우에는 적절치 못하다는 것이다. 이에 대응하기 위해서는 
지속적인 빌드 작업이 요구되어 진다.

<Callout>
  **장점**  
  1. 웹 사이트가 빠르게 동작한다.
  2. SEO 성능이 좋다.

  **단점**
  1. 동적인 웹사이트에는 적절치 못하다.
  2. 지속적인 빌드 작업으로 인해 시간이 많이 소요된다.
</Callout>
